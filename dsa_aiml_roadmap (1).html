The 5-step problem method
For slow learners who struggle with problem statements — read this before solving any problem
🎯 This is the most important tab in the entire plan
Most people who "can't do DSA" are not bad at coding. They're bad at reading problem statements. The problem statement is written in a very specific way. Once you learn to decode it — every problem becomes much easier. This 5-step method works on every DSA problem ever written.
1
Read once — just understand what is being given and what is wanted
Don't try to solve. Just identify:
→ What is the INPUT? (What data am I given?)
→ What is the OUTPUT? (What do I need to return?)
→ Read the examples first — they are clearer than the problem text.
Problem: "Given an array of integers, return the largest element."
INPUT: array of integers → [3, 1, 7, 2, 5]
OUTPUT: one number → 7
2
Write out the example by hand on paper
Copy the example from the problem onto paper. Trace through it manually.
→ What happens step by step to get from input to output?
→ This gives you the algorithm before you write any code.
→ If there are multiple examples — do ALL of them by hand.
Input: [3, 1, 7, 2, 5]
Step 1: Start with max = 3
Step 2: 1 < 3 → max stays 3
Step 3: 7 > 3 → max = 7
Step 4: 2 < 7 → max stays 7
Step 5: 5 < 7 → max stays 7
Output: 7 ✓
3
Write the algorithm in plain English — no code yet
Translate your paper trace into steps. Use words you'd use to explain to a friend.
→ "Start with the first element as the max"
→ "Loop through each remaining element"
→ "If current element is bigger than max, update max"
→ "Return max at the end"

If you can explain it clearly in words — the code writes itself.
4
Translate each English step directly to Python
One English step = one or two lines of Python. Nothing more.
Don't write the whole function at once. Write one step, test it, then the next.
def find_max(arr):
  max_val = arr[0] # Start with first element
  for num in arr: # Loop through each element
    if num > max_val: # If bigger than current max
      max_val = num # Update max
  return max_val # Return result
5
Test with the given examples AND your own edge cases
Run the examples from the problem — do they match expected output?
Then test edge cases:
→ Empty array → what happens?
→ Array with one element → correct?
→ All negative numbers → still works?
→ All same numbers → correct?

Edge cases are how interviews catch you. Testing them manually makes your solution bulletproof.
The 20-minute rule for slow learners: If you're stuck on a problem for 20 minutes — stop trying to solve it. Instead: look at the solution, understand it fully, then CLOSE IT and rewrite it from scratch without looking. That process = you actually learned it. Just reading solutions = you learned nothing.
How to decode confusing problem statements
When confused:Read the examples FIRST, not the problem text. Examples always tell you more.
Jargon:"Contiguous subarray" = a slice of the array with no gaps. "Subsequence" = elements in order but not necessarily adjacent.
Constraints:n ≤ 10⁵ means your solution must be fast. n ≤ 20 means brute force is okay.
"Return -1 if not found":This tells you the default case when there's no answer.
Still confused:Google "[problem name] explained for beginners" — someone has broken it down simply.

# Two pointer — find pair with target sum
def two_sum(arr, target):
    seen = {}
    for i, num in enumerate(arr):
        complement = target - num
        if complement in seen:
            return [seen[complement], i]
        seen[num] = i
    return []

# Binary search pattern — memorize this structure
def binary_search(arr, target):
    left, right = 0, len(arr) - 1
    while left <= right:
        mid = (left + right) // 2
        if arr[mid] == target: return mid
        elif arr[mid] < target: left = mid + 1
        else: right = mid - 1
    return -1

Revision strategy for slow learners
How to make sure you actually remember what you learned
🔁 The spaced repetition rule for DSA
Most people solve a problem, feel good, and never look at it again. A week later they can't solve it. The fix: revisit every solved problem 3 times — after 1 day, after 1 week, after 1 month. At each revisit: solve from scratch without notes. This is how you actually learn DSA, not just complete it.
The 3-revisit system
1st
Day after solving — quick revisit (10 min)
Look at the problem title only. Can you recall the approach? If yes — write it in 2 lines. If no — re-read your solution.
2nd
1 week later — solve again (20 min)
Open the problem. Solve it fresh. No looking at previous code. Time yourself.
3rd
1 month later — final check (15 min)
Solve one more time. If you can do it in 15 minutes — this problem is permanently learned. Mark it done.
Weekly revision block (Saturday 30 min)
10 minPick 2 problems from this week. Read only the title. Can you recall the pattern? Write it down.
10 minWrite in notebook: what patterns did I learn this week? Array problems use ___, Hashmap problems use ___.
10 minResolve your weakest problem from the week completely from scratch.
Pattern recognition notebook
Keep a running list of patterns. After solving any problem, add one line:
"Two Sum" → hashmap to store seen numbers
"Find max subarray" → sliding window or Kadane's algo
"Is palindrome" → two pointer from both ends
"Level order tree traversal" → BFS with queue
"Binary search" → sorted array, eliminate half each step
After 80 problems you'll have 80 patterns. Most new problems will match one you've seen.
The only metric that matters: Can you solve the problem from scratch one week later without looking at your notes? If yes — you learned it. If no — you just completed it. Those are very different things.
